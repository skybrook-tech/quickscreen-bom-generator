Background

Two BOM calculation implementations exist in the codebase, developed in parallel:

  • Data-driven engine — a Supabase Edge Function (bom-calculator) that reads rules, selectors, and companion rules from seeded Postgres tables and evaluates them at runtime using math.js. All calculation logic lives in JSON seed files per product.
  • Sandbox local calculator — a client-side TypeScript implementation (src/lib/localBomCalculator.ts) built as a public/offline fallback. Hardcodes BOM logic directly in TypeScript.

Several calculation decisions differ between the two. This report surfaces those differences so they can be resolved deliberately. UX/UI decisions will be consolidated separately once formula decisions are settled.

Formula Differences

The sandbox was written by the subject matter expert. These are cases where both implementations produce a number but use different logic. Neither is labelled correct here — these need resolving against the physical product.

1. Slat Count

The most consequential divergence. Results in a 1-slat difference in most real-world configurations, which cascades into actual height, stock lengths, and pack counts.

ApproachFormulaSlat widthRoundingResult (1800mm / 5mm gap)Actual height
Sandbox (SME)floor((H + gap - 3) / (w + gap))65.3mm physicalfloor25 slats~1748mm
Data-drivenceil((H - gap) / (w + gap))65mm nominalceil26 slats~1818mm

The SME used 65.3mm (the physical slat face width) and floor (slats that fit within the target). The engine uses nominal 65mm and ceil (slats needed to reach or exceed the target).

Which is correct: the number of slats that fit within the target height (SME / floor), or the number needed to reach or exceed it (engine / ceil)? Note that a third option — ceil with the SME's 65.3mm physical width — is not what either currently does.

2. Panel Width — Post Material Deduction

ApproachFormulaPanel width (10m run, five 50mm posts, 4 panels)
Sandbox (SME)segmentWidth / numPanels2500mm
Data-driven(segmentWidth − posts × postSize − wallDeductions) / numPanels~2437mm

The SME does not subtract post width from the run. The engine deducts post material first, reasoning that QSHS slats face-mount onto posts (clear face-to-face span).

Does the QSHS slat measurement convention include or exclude the post face width? This determines which is correct.

3. Spacer Quantity

ApproachFormulaSpacers per panel (26 slats)Packs (4 panels)
Sandbox (SME)2 × (n + 1) × panels / 50545 packs

The SME uses n + 1, implying spacers are also placed above the top slat and below the bottom slat. The engine uses n - 1, counting only gaps between slats.

Are spacers required at the top and bottom of the panel assembly, or only between slats?

Full Feature Diff

Beyond the formula differences above, the two implementations differ in product support and feature coverage.

Present in Sandbox (SME) — Missing from Data-Driven Engine

Economy Slats (finish_family = "economy")

The sandbox has a full economy finish path producing XP-6500-E65-{colour} SKUs (6500mm stock, no centre web). The engine's finish_type variable only offers standard and alumawood — no economy option exists.

Separate Post Colour

The sandbox reads post_colour_code as a distinct variable from colour_code, so a job can have a black fence with monument posts. The engine has no post_colour_code variable — posts always use the fence colour.

Colour Fallback for Non-Standard Colours

The sandbox explicitly handles colours outside the standard Colorbond range — CSR caps, CSR plates, and post accessories all fall back to MN (Monument) if the colour isn't supported for that accessory. The engine uses {colour} placeholders directly; an unsupported colour generates an invalid SKU with no fallback.

Installation Suggestions

The sandbox generates a separate suggestions pass with human-readable reasons for each item. None of these exist in the engine seed data:

SuggestionTrigger
Rapid set concrete (1.5 bags/post)mounting_type = in_ground
Core drill groutmounting_type = core_drill
Masonry anchor packmounting_type = base_plate
Touch-up paint (per colour used)Always

"Add to BOM" Suggestion UX

The sandbox has a fully built suggestions panel — each suggestion shows a reason and has an "Add to BOM" button. The engine has no equivalent UI.

Present in Data-Driven Engine — Missing from Sandbox

XPL Product

The sandbox explicitly does not support XPL and emits a warning for any XPL run. The engine has full XPL support including its proprietary post system (end posts, inline posts, corner posts) and 24mm slat insertion depth.

Gates (QS_GATE)

The sandbox always returns an empty gate BOM with a TODO warning. The engine handles gates fully.

VS Centre Support Rail

For VS panels wider than 2000mm the engine automatically adds a CSR with caps and top/base plates. The sandbox VS implementation does not emit CSR at all.

Corner Adapter (XP-6000-135)

The engine emits a corner adapter SKU when a panel boundary angle is ~135°. The sandbox does not handle corner adapters.

Structured Warnings and Blocking Validations

The engine has seeded per-product warnings (e.g. "panel over 1800mm — recommend HD post", "Primrose/Paperbark not available in 65mm HD post") returned as structured warnings[], errors[], and assumptions[]. The sandbox only emits ad-hoc warning strings.

Admin Trace Panel

The engine returns a full trace of which rules fired and what values were computed at each step — visible to admin users. The sandbox has no equivalent.

Suggestions Logic

Both implementations have a concept of "suggested accessories" — items shown to the user that are not included in the hard BOM until explicitly added. They handle this differently.

Sandbox — Dedicated Suggestions Engine

A standalone suggestAccessories() function runs as a separate pass after the BOM is calculated. Each suggestion carries a typed reason string and a priced flag, displayed in a dedicated UI panel with an "Add to BOM" button per item.

TriggerSuggested itemReason shown to user
mounting_type = in_groundRapid set concrete (1.5 bags/post)"Suggested at 1.5 bags per concreted-in post"
mounting_type = core_drillDress ring (XP-DR / XP-65DR)"Dress rings suit core-drilled posts"
mounting_type = core_drillCore drill grout"Choose grout product to suit substrate"
mounting_type = base_plateBase plate set"Base plate sets suit base-plate-mounted posts"
mounting_type = base_plateMasonry anchor pack"Select fixing type to suit substrate"
AlwaysTouch-up paint (per colour)"Suggested for colour-matched touch-ups after cutting"

The sandbox only suggests base plate sets and dress rings if they are not already in the hard BOM — correct deduplication behaviour.

Data-Driven Engine — Suggestions via Companion Rules

Certain companion rules in the seed data carry an is_suggestion: true flag. The engine emits these as a separate suggestions[] array. Currently only one item type is flagged as a suggestion across all products:

ProductSKUQuantityNotes
QSHS, VS, BAYG, XPLXP-BTP-{colour}2 per CSR per panelCSR top/base plates

The engine does not currently generate installation-related suggestions (concrete bags, masonry anchors, grout, touch-up paint). These are not in any seed file.

UX/UI Consolidation

The two implementations have developed UI components independently. These decisions have not yet been consolidated and should be addressed as a separate workstream once formula decisions are settled. Key areas requiring consolidation:

  • BOM results display — both have a BOMResultTabs component but with different tab structures and category ordering
  • Suggested accessories panel — the sandbox has a full SuggestedAccessoriesPanel with add/added state; the data-driven engine surface needs to be designed
  • Warnings and assumptions display — the data-driven engine has BOMWarningsPanel with structured severity levels (warning, info, error); the sandbox mixes these into a single list
  • Achieved height badge — exists in the data-driven engine's UI (AchievedHeightBadge); not present in the sandbox
  • Admin trace panel — data-driven engine only, gated by user role
  • Public vs authenticated experience — the sandbox was explicitly built for an unauthenticated route; the data-driven engine requires login. This is the primary driver of the split and the UX implications of any consolidation decision flow from it

Structural Pros and Cons

Data-Driven Engine

Pros

  • Product-agnostic — new fencing system = one JSON file, no code changes
  • All behaviour visible and editable in seed files without a deployment
  • Formulas are version-controlled and testable in isolation
  • Supports all products including XPL and gates
  • Fixture suite provides regression coverage
  • Structured warnings, info messages, and blocking errors per product
  • Admin trace output shows exactly which rules fired and why

Cons

  • Requires live Supabase connection — no offline/public access without auth changes
  • Debugging requires understanding the math.js rule evaluation pipeline
  • Formula changes require a DB seed update and reset cycle
  • Installation-related suggestions not yet implemented
  • Some edge-case colour fallback logic less explicit than sandbox

Sandbox Local Calculator

Pros

  • Works without authentication — usable as a public calculator
  • All logic in one readable TypeScript file — easy to trace
  • Richer installation suggestions with human-readable reason text
  • Full "Add to BOM" suggestion UX already built
  • Colour fallback for non-standard colours more explicitly handled
  • 65.3mm physical slat width may be more accurate to supplier specs

Cons

  • Three confirmed formula divergences from fixture suite
  • XPL and gates not implemented
  • VS missing centre support rail
  • Any formula change requires a code deployment
  • Duplicates logic from seed data — two sources of truth to maintain

AI Integration Comparison

Importing New Fencing Systems

The seed format is structured JSON with a defined schema (supabase/seeds/schemas/). Adding a new fencing product means producing one JSON file that describes variables, rules (math.js expressions), component selectors, companion rules, and pricing. A seed-mapper skill already exists in the codebase to guide this process.

Data-Driven Engine

  • AI reads a supplier PDF or build pack and extracts values into the JSON format
  • Schema validates the output automatically on npm run seed:products
  • A bad seed file fails validation and is rejected cleanly — can't corrupt existing products
  • AI doesn't need to understand the engine pipeline — only the schema
  • In practice: supplier spec sheet in → draft seed file out in one pass, human validates against a known-good job

Sandbox Calculator

  • Adding a new product requires modifying TypeScript logic — new branches, SKU builders, colour constants
  • No schema to validate against — errors surface at runtime
  • Each new product increases code complexity and maintenance burden
  • A bug silently produces wrong quantities rather than failing loudly
The data-driven engine's structured JSON format is essentially an AI-friendly interface to the calculation engine. The seed-mapper skill is already in place; the next step is testing it end-to-end against a real supplier document.

Using AI to Operate the Calculator (Job Description → BOM)

This is a planned feature — natural language job descriptions parsed into a canonical payload, then run through the calculator.

Example flow (data-driven engine):

"10 metre black monument fence, 1800mm high, base plated, 2 gates"
→ AI produces CanonicalPayload JSON
→ POST /functions/v1/bom-calculator
→ Full priced BOM returned

The AI never touches the calculation logic — it only handles the natural language → structured data conversion. The AI's output is validated against the Zod schema before the calculator is even called. This separation is clean and the task is a reliable, constrained extraction.

The sandbox accepts the same canonical payload format, so the same AI parse step would work in principle. However, its formula gaps (XPL, gates, divergent slat count) mean the AI-generated payload may be correct while the resulting BOM is wrong. Running AI API calls client-side also has security implications.

AI BOM review (also planned) is only meaningful against a BOM you trust. The data-driven engine's trace output — which rules fired and why — provides a much better foundation for an AI reviewer than the sandbox, which has no equivalent trace.
