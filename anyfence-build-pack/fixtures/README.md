# Golden BOM Fixtures

These are **`input → expected` regression tests** generated by running the real reference engine (`../skills/calculator-engine/treated-pine-paling-fence-calculator/calculator.py`). They are the contract for the treated-pine paling calculator.

## How Antigravity should use them

1. Re-implement the calculator in the target stack (TS/Node, etc.).
2. For each `*.input.json`, run your engine and assert the output **equals** the matching `*.expected.json`.
3. Keep them as regression tests. If you intentionally change engine behaviour, regenerate with `python3 _generate_fixtures.py` and review the diff — and if any `canonical_name` string changed, **bump the canonical-name contract version** (see `../knowledge-pack/04-canonical-product-names.md`).

Each `*.expected.json` is the full engine result: `{ ok, inputs, derived, bom, warnings, constants }`. `manifest.json` summarises all cases.

## Cross-checks worth knowing (verify your reimplementation matches)

- **Paling rate:** butted = 10/m (20m → 200 palings); lapped & capped = 15/m (20m → 300 palings). Then +5% wastage, `ceil`.
- **Posts:** `ceil(length_mm / post_spacing_mm) + 1` (20m @ 2400 → 10 posts, 9 bays).
- **Rails:** `bay_count × rail_count`, then `ceil(pieces / 2)` stock lengths (4800mm stock, 2 pieces each).
- **Rail count by height:** 1200→2, 1500/1800/2100→3, 2400→4.
- **Nails:** butted = palings × rails × 2 (57mm ring-shank), +5% wastage, rounded up to nearest 250-pack. Lapped adds a back layer at 45mm × 1/rail.
- **Concrete:** `ceil(post_count × 1.5)` bags (20kg Rapid Set).
- **Posts run 600mm into the ground:** `post_height = fence_height + 600`.

## ⚠️ Known engine hardening note (FIX in the reimplementation)

`calculate_bom()` extracts `inputs["fence_length_m"]` and `inputs["fence_height_mm"]` with **direct `[]` access *before* `validate_inputs()` runs**. So when a required field is **missing entirely**, the engine raises an unhandled `KeyError` instead of returning a clean `{ ok: false, errors: [...] }`. (Invalid-but-present values like `post_spacing_mm > 2400` are handled correctly — see the error fixture.)

This is fine for a trusted CLI but **not safe for a public API / marketplace endpoint** that receives arbitrary input. In the reimplementation: **validate first, return the error envelope on failure, then extract** — so every bad input yields a structured error, never a 500. (This is why the error fixture uses an out-of-range value rather than a missing field.)
