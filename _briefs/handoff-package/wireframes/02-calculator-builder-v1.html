1 Open the Builder

First impression: Build Forge says hi and asks one question.

The supplier hits /builder and Build Forge introduces itself. No form fields, no setup wizard, no settings tabs — just a chat with one decision: tweak an existing fence type, or build from scratch. The preview pane is empty for now, with a polite "your calculator will appear here as we build it."

What's on screen

  • Anyfence brand top-left, supplier name + logo as they're set (here: "New calculator" because not yet configured)
  • Progress bar with 6 steps — first step active, rest greyed
  • Left: Chat panel with Build Forge. Its first message is short, friendly, sets the time expectation (~20 min)
  • Right: Preview pane with empty-state message
  • Quick-reply chips below Build Forge's message — primary CTA in ember, secondary in white

Two paths, picked by the supplier

  • Tweak existing (default, recommended) → next stage shows archetype picker (Timber Paling, Colorbond, Pool Glass, …)
  • Build from scratch → longer flow, ends with new archetype proposal that Fence Forge reviews before going live

Tone

  • Plain, friendly, time-bounded (the "20 minutes" promise is a feature)
  • "BF" avatar — Anyfence-ember coloured, lower-case feel, like Notion or Linear's AI
  • No jargon yet. No "canonical names", no "archetypes", no "instances"

3 Upload catalogue

"Drop your price list and I'll work out what's in it."

Build Forge needs the supplier's products + pricing. It accepts everything: Cin7 xlsx, generic CSV, supplier PDF, photo of a price sheet, or a link to their online catalogue. The upload zone lives inside the chat. Background processing happens visibly.

Accepted inputs

  • xlsx — Cin7 mass-download, generic Excel
  • csv — plain text price lists
  • PDF — supplier brochures or printed price sheets
  • Photo — phone snap of a paper price list (OCR via Build Forge)
  • URL — link to an online catalogue (Bunnings, supplier site)
  • Manual — type/paste rows one at a time

What Build Forge does behind the scenes

  • If xlsx: inspects headers, identifies the price column (BuyPriceEx for tier-2 trade, RetailPrice for tier-1, etc.)
  • If PDF: extracts tables, falls back to OCR if needed
  • If URL: uses BrowserSession + bunnings-fence-scraper patterns
  • Always shows progress — never leaves the supplier staring at a spinner

Engine binding

  • Output is a structured catalogue rows array — supplier_id × canonical_name (proposed) × supplier_sku × price × pack_size
  • Rows tagged with confidence: high (clean match) / medium (best guess) / low (needs human review)
  • Goes into stage 4 — mapping — for explicit confirmation

4 Map catalogue

"Here's how I've matched your products to fence names. Anything wrong?"

The translation moment. Build Forge has parsed the xlsx and proposes a mapping of supplier SKU → canonical product name. The supplier confirms in bulk for the high-confidence ones, drills into anything flagged. Gaps in the catalogue (missing canonical names) get explicit attention.

Why this stage is critical

  • The canonical-name contract is everything — wrong mapping = wrong BOM forever
  • Build Forge shows confidence per row so the supplier focuses attention
  • High-confidence rows default to confirmed; supplier can object
  • Medium rows need a tap to confirm
  • Low / gap rows pause the flow until resolved

"NEW canonical name" handling

  • If a supplier product doesn't match any existing canonical name (e.g. "120×16 Treated Pine Paddle Paling"), Build Forge proposes a NEW canonical name
  • The proposal needs the supplier's confirmation AND gets flagged for Fence Forge review
  • Build Forge never silently invents canonical names — they're a versioned contract
  • The amber "NEW" badge in the mapping table is the visible signal

Engine binding

  • On confirm: writes to system_instances.catalogue (per migration 032)
  • New canonical names: drafted in canonical_names_proposals for Fence Forge to merge
  • Each row carries price_book_version so the eventual quote can pin to it

5 Variations + the magic translation

"How many heights do you sell?" Then: "I do palings on a 45 diagonal."

Two things happen in this stage. The easy one — build up the variation form by asking conversationally. The hard one — when the supplier describes a custom layout (palings sideways, on a diagonal, board-on-board), Build Forge translates that to a BOM formula and shows it back in plain English plus a worked number. The whole "20 minutes for full power" promise lives here.

Variations Q&A pattern

  • One question, multiple-choice chips for the easy answers
  • Supplier can pick multiple chips for "we offer all of these"
  • Each answer narrows the next question — heights inform paling-length defaults, etc.
  • Live preview updates the variation form on the calculator in real time

The "diagonal palings" moment

  • Supplier says it in plain English: "I do palings on a 45 diagonal"
  • Build Forge asks ONE clarifying question (do palings still butt edge-to-edge, or with a gap?)
  • Translates to a formula: cuts_per_run_fn = ceil(run_length_mm / (paling_width_mm × cos(45°) - 6))
  • Shows the formula AND its plain-English meaning AND a worked example ("for a 10m fence, that's 98 palings instead of the 70 you'd use butted")
  • Supplier confirms or says "no, that's too many — let me show you how I actually do it"

Why this is the leverage moment

  • This is the bit that makes "non-technical + full power" actually work
  • Without the AI translation, a wizard tops out at "pick variations from a fixed list"
  • With it, suppliers describe ANY layout they actually use — even weird ones — and Build Forge figures out the math
  • If Build Forge can't figure it out, it surfaces to Fence Forge (don't silently guess)

6 Test + publish

Build Forge runs a worked example, then hands over the preview URL.

Before going live, Build Forge runs a worked example BOM and shows it for confirmation. Compliance checks fire here: H3/H4 timber treatment, council height caps. Once the supplier signs off, the publish flow opens — draft → preview URL → publish, with a price-book version pinned.

Test step — the trust moment

  • Build Forge picks a typical job dimension (12m × 1800mm) and runs the BOM against the new config
  • The supplier sees a full quote (canonical names, supplier SKUs, quantities, prices, total)
  • "Is this what you'd quote for this job?" — if no, drills into the lines that differ
  • Builds confidence that the math is right BEFORE going live

Compliance gating

  • Build Forge runs through applicable rules: H3/H4 timber treatment, council height caps for the supplier's state, neighbour-consent reminders
  • For pool/timber/bushfire — surfaces relevant block_quote rules
  • Supplier can't publish without acknowledging applicable rules (audit-logged)

Publish flow

  • Draft → preview URL (private, shareable for supplier's internal review)
  • Supplier shares with their team if they want
  • Then publish — calculator goes live at /s/amazing-fencing/calculator/timber-paling
  • Price-book v1 pinned · future edits create v2 etc., quotes pin to whichever was live at quote time

What's deliberately not in this wireframe
The "Build from scratch" path

This wireframe shows the Tweak existing path. "Build from scratch" — for genuinely new fence archetypes — is a separate flow with more compliance gating, fixture authoring, and a Fence Forge review step before going live. Wireframe v2.

Edit-an-existing-calculator flow

After first publish, the supplier comes back to /builder/timber-paling to tweak. That UI is mostly the same as this one but with everything pre-filled and the chat focused on "what would you like to change?" — a thinner walkthrough.

Mobile, multi-user, supplier dashboard

Mobile breakpoint stacks chat above preview. A supplier dashboard at /builder/dashboard lists all their published calculators with edit/preview/duplicate actions. Separate wireframe.
